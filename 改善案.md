# 湊 質問AI — 改善案メモ

## 🔴 優先度高：セキュリティ・乱用対策

### 1. APIが誰でも叩ける状態（最重要）
- `api/chat.js` は CORS が `*` で認証なし。URLさえ知っていれば**塾生以外でも無料でGeminiを使い放題**になり、API料金が膨らむリスクがある。
- 対策案（軽い順）:
  - CORSを自分のドメインだけに限定（`Access-Control-Allow-Origin` を本番URLに）
  - 塾生共通の「合言葉（アクセスコード）」を初回モーダルで入力させ、サーバー側で照合
  - 将来的には生徒ごとの簡易ログイン

### 2. レート制限がVercelでは実質効かない
- `requestLog` はメモリ上のMap。Vercelのサーバーレス関数は**インスタンスが複数立ち上がり、すぐ破棄される**ため、カウントが共有されず制限が抜けられる。
- キーが `studentName` なので、名前を変えれば無制限に回避可能。
- 対策案: Upstash Redis（無料枠あり）や Vercel KV でカウントを共有し、キーは「名前+IP」にする。

### 3. 入力の検証が緩い
- `message` の文字数上限なし → 超長文を送られるとトークン代が跳ねる。例: 2000字で切る。
- `history` はクライアントから送られたものをそのまま信用 → 件数・各テキスト長を**サーバー側でも**制限する（現状はフロントの `MAX_HISTORY=6` 頼み）。
- `imageMimeType` のホワイトリスト検証なし（`image/jpeg`, `image/png` 等のみ許可に）。
- 画像base64のサイズ上限チェックなし（Vercelのボディ上限4.5MBに達すると不親切なエラーになる）。

### 4. Apps Script のログURLも誰でもPOST可能
- 「全員アクセス可」のWebアプリなので、URLが漏れるとスプレッドシートにゴミ行を書き込まれる。
- 対策案: ペイロードに共有シークレット（例: `data.token`）を含め、`log.gs` 側で照合して不一致なら破棄。

## 🟡 優先度中：応答品質・コスト

### 5. thinking トークンの無駄（gemini-2.5-flash）
- 2.5-flash はデフォルトで「思考（thinking）」が有効。中学生向けの短い返答には過剰で、**レイテンシと出力トークン代が増えている**可能性が高い。
- `generationConfig` に `thinkingConfig: { thinkingBudget: 0 }` を追加して比較テストする価値あり（数学の難問だけ思考ありにする二段構えも可）。

### 6. maxOutputTokens 1024 は目標（150字）に対して大きい
- thinkingを切るなら 512 程度でも足りるはず。切らない場合はthinking分を含むので注意。

### 7. 画像つき質問の文脈が続かない
- `history` にはテキストしか入れていないため、「この写真の問題の続きだけど…」という**2ターン目以降でAIが画像を忘れる**。
- また画像のみ（テキストなし）の質問はフロントで `history` に積まれず、会話が欠落する。
- 対策案: 直近1枚だけ画像もhistoryに含めて再送する／「（画像で質問した）」というプレースホルダーテキストを積む。

### 8. Markdownが生テキストのまま表示される
- 回答に `**太字**` や `- 箇条書き` が混ざるとそのまま記号が見える。
- 対策案: プロンプトに「Markdown記法（**や#）を使わない」と明記する（一番簡単）、またはDOMPurify+marked で安全にレンダリング。

### 9. プロンプトの回帰テストがない
- プロンプトを直すたびに手動で試している状態。「答えを丸ごと聞く」「雑談を振る」「式を打たせようとするか」など**定番テスト質問集（10〜20問）**を作り、変更後に一括で流して比較する仕組みがあると安心（スクリプト1本で可）。

## 🟢 優先度中〜低：UX改善

### 10. 応答のストリーミング表示
- 現状は全文生成を待ってから表示。`streamGenerateContent` + SSE にすると体感がかなり速くなる。

### 11. リロードで会話が消える
- `history` はJS変数のみ。誤リロードで全部消える。名前と同様 `sessionStorage` に保存すれば復元できる。

### 12. タイムアウト・再送がない
- fetchにタイムアウト（例: 30秒で `AbortController`）と、エラー時の「もう一度送る」ボタンがあると生徒が詰まらない。

### 13. フィードバックボタン
- AI回答に「👍わかった / 👎まだわからない」を付けてログに記録すると、**どの教科・単元でAIが役に立てていないか**が数字で見える。プロンプト改善の材料として最も費用対効果が高い。

### 14. 生徒名の表記ゆれ（model_notes.md にもあるオートコンプリート案）
- ログ集計の妨げになるので、名簿スプレッドシート＋候補表示の折衷案は早めに価値あり。合言葉（改善案1）と組み合わせられる。

## ⚙️ 運用まわり

### 15. ログ送信で応答が遅くなっている
- Apps Script への送信を `await` しているため、**生徒への返信がログ送信の分（数百ms〜数秒）遅れる**。
- `@vercel/functions` の `waitUntil()` を使えば、レスポンスを先に返してからバックグラウンドでログ送信できる。

### 16. コスト監視
- トークン数はログにあるので、スプレッドシートに「日別合計トークン・推定料金」の集計シート（QUERY関数）を1枚足すだけで見える化できる。Google Cloud側で予算アラート設定も推奨。

### 17. .gitignore がなかった
- 今回作成済み（このメモ、`.env*`、`node_modules/`、`.vercel/` など）。APIキーを誤ってコミットする事故防止。

### 🆕 2026-07-16〜17に新規実装した運用機能
- **週次利用状況の自動記録**（`log.gs`）：毎週月曜の朝、先週分（月〜日）の生徒ごとの質問回数・合計Tokenを「週次利用状況」シートの先頭に自動追記（新しい週が上）。誰がよく使っているかを把握する目的。
- **質問ログ・フィードバックログの「新しい順」ビュー**：元データの書き込み順（下に追記）は変更せず、SORT関数で参照する別シートを追加。書き込み速度への影響なし。
- **clasp導入**：Google Apps ScriptプロジェクトをCLIから直接push/deployできるようにした。今回の作業で、実は`log.gs`の一部の修正（共有シークレット対応・フィードバックToken記録）が本番のApps Scriptデプロイに反映されていなかった（手動デプロイ漏れ）ことが判明し、あわせて同期・修正した。

### 🆕 未解消の軽微な既知事項（実害は小さいが記録として残す）
- **フィードバックの科目判定コストが集計に含まれていない**：`api/feedback.js`は👍👎のたびにGeminiへ科目判定の軽いリクエストを送るが、そのトークン数は「集計」シートの日次コストには反映されない。1回あたり非常に軽い（`maxOutputTokens:10`）ため実害は小さい。
- **`log.gs`の`LockService`ロックが、通常ログとフィードバックログで共有**：`doPost`全体を1つのロックで囲んでいるため、生徒の質問ログとフィードバック送信が同時に来ると片方が待たされる。実際の生徒数ではまず問題にならない規模。書き込み先シートが別なので、本来は別々にロックしても支障はない（意図的に現状維持と判断済み）。
