## 1. これは何か

塾の生徒が、スマホ・タブレットから勉強の質問を送ると、AI（Google Gemini）が家庭教師のように答えてくれるWebチャット。
質問ボックスの代わりとして、生徒がいつでも質問できることを目的にしている。

- 想定ユーザー: 中学生（中1〜中3）が中心。高校生も利用可（学年は分けず「高校生」）。各自が自分の端末で使う。
- 公開URLのWebサイト（Vercelでホスティング）。

## 2. 技術構成・ファイル

| 部分 | 技術 | 役割 |
| --- | --- | --- |
| フロント（画面） | HTML/CSS/JS 1ファイル | `public/index.html`。チャット画面、名前/学年/クラス入力、画像送信、KaTeXで数式表示 |
| バックエンド（中継） | Vercel Serverless Function | `api/chat.js`。Gemini APIへ質問を中継し、回答を返す。ログも送る |
| ログ保存 | Google Apps Script + スプレッドシート | `log.gs`。質問・回答などを1行ずつ記録 |

使用モデル: **`gemini-2.5-flash`**（Gemini API）。

APIキーなどの秘密情報は **Vercelの環境変数** に入れる（コードに直書きしない）。
- `GEMINI_API_KEY` … Gemini APIのキー
- `LOG_WEBHOOK_URL` … Apps ScriptのウェブアプリURL（`/exec` で終わるもの）

**注意**: `log.gs`の`SHEET_NAME`定数は`'Sheet1'`（英語）だが、実際のスプレッドシートのタブ名は`シート1`（日本語）。一致しないため`getSheetByName('Sheet1')`は常に失敗し、`getActiveSheet()`のフォールバックで動作している（実害はないが、集計シートの数式などでシート名を書くときは実際のタブ名`シート1`を使うこと）。

## 3. いまある機能

### 生徒の画面
- **名前・学年・クラスの入力**: 最初にモーダルで入力。学年は中1 / 中2 / 中3 / 高校生。
  - 中3 → クラス `S（御三家志望）` / `A` / `B` / `個別`
  - 中1・中2 → クラス `A` / `B` / `個別`（Sは中3だけ表示）
  - 高校生 → 学年は細かく分けず「高校生」の1択。クラスなし（クラス欄は自動で隠れ、選択不要）。先取り学習の子がいるため学年で範囲を縛らず、AIは質問内容からレベルを判断する。
  - 右上の「名前変更」でいつでも変更可。
- **入力の保持は sessionStorage**: タブを開いている間とリロードでは名前を覚えているが、**タブを閉じると次回は必ず聞き直す**。共用端末でも他人の名前が固定されないようにするため。
- **会話履歴もsessionStorageに保存**: ページをリロードしても会話が消えず復元される。「名前変更」で別の名前に変えたときは、前の生徒の会話が見えないよう履歴・画面ともクリアされる（同じ名前のまま学年・クラスだけ更新した場合は履歴を保持）。画像のみ（テキストなし）の質問も「（画像で質問した）」というプレースホルダーで必ず履歴に積まれる（以前は欠落していた）。
- **テキスト質問**＋**画像送信**（カメラ撮影・アルバムの写真どちらも選べる）。
- **数式表示**: AIの回答内の数式はKaTeXできれいに表示（`$...$`、`$$...$$`）。長い数式（`$$...$$`）は吹き出しの幅を超える場合、数式部分だけ横スクロールできるようにして、画面全体がはみ出さないようにしている（`.katex-display { overflow-x: auto }`）。
- **太字・ハイライト表示**: AIの回答内の`**太字**`は実際の太字に、`==ハイライト==`は水色の背景色付きバッジに変換して表示（見出し・箇条書きは使わない方針。詳細は「4. AIへの指示」参照）。innerHTMLは使わず要素を組み立てて描画するのでXSS対策も維持。
- **クイックアクションボタン**: 「🍀 やさしく」（直前の説明をもっとやさしく短く言い直してもらう）／「⏩ 次のステップ」（もう分かっている内容を繰り返させず、手短に次へ進めてもらう）／「🆕 次の問題」（画面の会話は残したまま、AIに送る文脈だけ区切る。区切り線「次の問題」を挿入し、以降のリクエストにはそれより前の会話を含めない）。
- **フィードバックボタン（👍/👎）**: AIの回答ごとに表示。押すと`api/feedback.js`に送信され、専用の「フィードバック」シートに記録される（詳細は下記バックエンド参照）。押した後はボタンが「ありがとう！」に切り替わる。
- **混雑時メッセージ**: レート制限（429）に当たると「1分くらい待ってね」と表示。
- **一時的な通信エラーの自動リトライ**: 502（Geminiとの通信エラー）やfetch自体の失敗時は、1.5秒待って最大2回まで自動で静かに再送する。生徒には気づかれずに復旧することが多い。400（入力不備）・429（混雑）は再送しても解決しないため対象外。全部失敗したときだけ「通信エラーです」と表示。
- **通信タイムアウト**: fetchは25秒応答がなければ`AbortController`で強制的に打ち切る（無応答のままローディング表示が終わらなくなるのを防ぐ）。打ち切りは通信エラー扱いとなり、上の自動リトライの対象にもなる。

### バックエンド（`api/chat.js`）
- フロントから `studentName / studentGrade / studentClass / message / 画像 / 直近履歴` を受け取る。
- AIへの指示（システムプロンプト）＋クラス別の指導方針を組み立ててGeminiに投げる。
- 直近の会話履歴を最大 **14件**（`MAX_HISTORY = 14`）まで一緒に送って文脈を保つ。フロントから送られてきた`history`はサーバー側でも件数（直近14件）・各エントリの文字数（2000文字）を超えた分を静かに切り詰める（`/api/chat`を直接叩いて大量・長文の履歴を送りつける悪用対策。フロントの制限だけに頼らない）。
- 429（混雑）/ その他エラーを生徒向けのやさしい文言にして返す。
- 質問文（`message`）は**2000文字**まで（フロントの`maxlength`とサーバー側チェックの両方）。超えると「質問が長すぎるみたい」エラーを返す。長文貼り付けによるコスト増・悪用を防ぐ目的。高校数学の長めの大問でも十分収まる余裕を持たせた文字数。
- 画像も`imageMimeType`をホワイトリスト（`image/jpeg`・`image/png`・`image/webp`）でチェックし、`imageBase64`は約4MB相当を超えたら拒否。フロントは常に1600pxにリサイズしてから送るので通常利用では引っかからないが、`/api/chat`を直接叩かれた場合の悪用・不正データ対策。
- CORSは本番サイト（`https://juku-ai-chat.vercel.app`）のみ許可。以前は`*`（全許可）で、他サイトに埋め込まれたJSから訪問者のブラウザ経由で無関係にAPIを呼ばれる余地があった。自サイトが`/api/chat`を呼ぶのは同一オリジンなのでこの制限の影響は受けない（プレビュー環境のテストも問題なし）。
- 回答が返ったらログをApps Scriptに送る（日時・学年・クラス・生徒名・質問・画像あり/なし・AI回答・トークン数・本日の累計Token）。画像そのものは保存しない。
  - スプレッドシートの列は A〜K: 日時・学年・クラス・生徒名・質問・画像・AI回答・入力Token・出力Token・合計Token・**本日の累計Token（K列、サイト全体の当日合計）**。生徒名の前に学年・クラスをまとめる並び順（2026-07-10時点）。
  - 既存シートの列順がこれと違う場合は、Google Sheets上で列を手動で並び替えてから使うこと（コードは常にこの並び順でA〜K列を扱うため、ズレたままだと違う列にデータが入る）。
  - `log.gs` はGitHub pushでは反映されない別経路。Apps Scriptエディタに貼り直して「デプロイを管理 → 編集 → 新バージョン」で再デプロイが必要。
  - ログ送信は`@vercel/functions`の`waitUntil()`でバックグラウンド化。生徒への返信はログ送信の完了を待たずに返る（無料プランの実行時間上限300秒に対し、ログ送信は数百ms〜数秒程度なので十分収まる）。
  - `log.gs`側は`LockService`で`doPost`をロックし、生徒が同時に質問しても書き込みが1件ずつ順番に処理される（同時アクセスで行がズレる・累計を誤った行に書き込む事故の防止）。

### バックエンド（`api/feedback.js`）
- フロントの👍👎ボタンから `studentName / studentGrade / studentClass / feedback(good|bad) / questionText / aiReply` を受け取る。
- 生徒の質問文からGeminiで教科（数学・英語・国語・理科・社会・その他）を自動判定（`temperature:0`・`maxOutputTokens:10`の軽いリクエスト）。
- 科目判定・ログ送信とも`waitUntil()`でバックグラウンド化し、生徒への応答（「ありがとう！」表示）は待たせない。
- ログは`log.gs`に`type: 'feedback'`として送られ、通常の質問ログとは別の「フィードバック」シート（日時・学年・クラス・生徒名・評価・科目・生徒の質問・AI回答・入力Token・出力Token）に記録される。シートが無ければ自動作成。科目判定リクエストのトークン数も記録するので、「集計」シートに含めれば正確なコスト把握ができる（現状「集計」シートは`シート1`のみ集計。含めたい場合は数式にフィードバックシート分を追加する必要あり）。
- `/api/chat`と同様、CORSは本番ドメインのみ許可。同一生徒からの連続送信は1分20回までにレート制限。

### コスト集計（スプレッドシート「集計」シート）
- ログシート（`シート1`）とは別に「集計」シートを追加し、今日・今週（月曜始まり）・今月の入力Token/出力Token/概算料金($)/概算料金(¥)を数式で自動集計。コード変更・再デプロイ不要（スプレッドシートの数式のみ）。
- 概算料金は有料プラン単価（入力$0.30／出力$2.50 per 1Mトークン）で計算。円換算は`GOOGLEFINANCE("CURRENCY:USDJPY")`で為替レートを自動取得。
- 数式は`シート1!A2:A9999`のように**2行目から**参照すること（1行目の見出しを含めると、見出しの文字列が数値計算に混ざってエラーになる）。日付の解析に失敗する行があってもエラーで止まらないよう`IFERROR`で包んである。

## 4. AIへの指示（プロンプト）の方針

実体は `api/chat.js` の `SYSTEM_PROMPT`（全生徒共通）と、`buildLevelInstruction()`（学年・クラス別に追加）。
中身を変えるときはこの2か所を編集する。

### 共通方針（SYSTEM_PROMPT）の要点
- **答えを丸ごと教えない**。ヒントと「次の一歩」で自分で解かせる。最後の答え合わせは生徒が出してから。
- **返信は短く**（3〜5行・150字程度）。一度に1ステップだけ。次は生徒が反応してから。
- **やさしい言葉**で。専門用語は避け、中学生の言葉で。
- **同じ説明を繰り返さない**: 同じ考え方の問題が続くときや、すでに理解を示した内容は、くわしく説明し直さない。
- **質問は控えめ**。いきなり質問返しせず、まず小さなヒント→最後に「はい/いいえ」で答えられる簡単な質問を1つだけ。
- **問題文を確認してから答える**: 小問が進むときや1つの設問に複数の問いがあるときは、答え始める前に読み取った問題文を書いて確認する。生徒が「違う」と言ったら、正しい問題文（テキストまたは写真）をもらってから続ける（問題の読み取りミス対策）。写真の問題は次のやり取りで写真を送り直さない前提のため、後で写真なしでも解き続けられるだけの情報を文章に書き起こしておく（画像つき質問が2ターン目以降で文脈から消える問題への対処）。
- **ゴールを見失わない**: 最初に確認した問題文＝最終的に出すべき答え。公式確認や用語理解はその途中でしかなく、それ自体をゴールにしない。数回やり取りが続いたら「もともと聞かれていたのは〜だったね」と目的を思い出させ、生徒が具体的な答えを出すまで会話を終わったとみなさない（やり取りの中でゴールがすり替わる対策）。
- **生徒に式を打たせない**。式はAIが書いて「合ってる？」と確認、または番号で選ばせる。書いた式を見たいときは写真を促す。
- **生徒が書いた式の並び順をそのまま扱う**: 「10a+b」のような教科書の定番形に勝手に書き換えない（例: 生徒が「10b+a」と書いたら「10b+a」のまま扱う）。
- **答え合わせは実際に確かめてから**: 生徒の式・数値が合っているかを見た目の類似だけで判断しない。具体的な数値を当てはめて計算するなど確実な方法で確認してから伝える（例: `b+10a`と`10a+b`は同じだが、`10b+a`とは別物。AIがこれを混同して誤って「同じ」と答えてしまった実例への対処）。
- **途中式の写真を要求**: 計算問題は、答え合わせの前に必ず一度「ノートの写真を送ってね」と求める（自分で解いた確認のため）。用語の質問では不要。
- **見出し・箇条書きは使わない**: 強調したい言葉は`**太字**`、特に大事な数値・結論は`==ハイライト==`で囲む（フロント側でそれぞれ実際の太字・色付きバッジに変換される）。
- **AIの限界を正直に**: 不確かなときは断定せず、難しければ「先生に聞いてね」と促す。
- **勉強以外・不適切な内容には応じない**。

### クラス別の指導方針（buildLevelInstruction）
| クラス | 方針 |
| --- | --- |
| S（中3・御三家志望） | ヒント最小限。自分で考えさせる時間を長く。本質や発展的な見方にも触れる。答えは丸ごと教えない |
| A（上位） | 基礎はできている前提。標準〜応用。ヒントやや少なめ |
| B（基礎） | 一歩を小さく、言葉を特にやさしく、例を多めに。よくほめる |
| 個別 | その子のペース最優先。つまずきを一つずつ丁寧に。ゆっくり |
| 高校生（クラスなし） | 学年で範囲を縛らず質問内容でレベル判断。発展的な内容にも対応。答えは丸ごと教えない |

加えて、中学生では学年（中1〜中3）が習う範囲を超えた難しすぎる解法は避けるよう指示している。

**高校生限定：手順の番号選択**（2026-07-14〜）。新しい問題に取り組み始めるとき、高校生にだけ「解くまでの大まかな手順を①②③…の番号付きリストで示し、生徒が番号でどこから聞きたいか選べる」ようにする指示を追加。すでにわかる手順を自分で飛ばせるようにする狙い。選んだ手順より前は省略してよいが、選んだ手順からは通常通り1ステップずつヒントで進め、答えを丸ごと教えない方針は変わらない。中学生には出さない（学習段階的に手順の全体像を先に見せるのは避けたいクラスがあるため）。新しいUI・構造化出力は不要で、既存の「番号で選ばせる」パターンの延長として実装（プロンプトのみの変更）。実際の生成品質はAPIキーでの確認が必要。

## 5. 今後の検討メモ（未実装）

- **教科ごとのルール・教え方**: 数学は途中式の写真必須、社会は写真不要、など教科別に対応を変える案。判別方法は「生徒が教科ボタンで選ぶ（確実）」か「AIが自動判別（手軽）」の2案。`buildSubjectInstruction(教科)` を足してプロンプトに合体させるイメージ。→ まずは現状のまま生徒に使ってもらい、反応を見てから判断する。
- **生徒名簿のオートコンプリート**: 名前の表記ゆれやクラス選び間違いを防ぐ案。ただし「全生徒一覧を公開URLで見せる」のはプライバシー上NG。名前を打つと候補が出て確定＋クラスは登録済みを自動入力、という折衷案が候補。名簿はGoogleスプレッドシートで管理してサイトが読み込む構成を想定。今は自由入力のまま運用中。
- 類題の出題、回答末尾の「今日わかったこと」まとめ なども会話で挙がったアイデア（未着手）。
